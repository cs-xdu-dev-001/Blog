# 首页移动端性能设计

## 目标

保留影像档案两排反向自动轮换及完整数据覆盖，同时显著降低手机和平板首页的DOM、图片解码、事件监听和后台动画开销。

## 已确认根因

- 首页把约360条影像拆成两行后各复制一次，实际生成720张影像卡。
- 390px和768px视口均包含约5850个DOM元素、731张图片和735个倾斜交互节点。
- 手机关闭影像轨道动画，但保留全部DOM、图片节点和监听器。
- 768px平板会持续移动两条超过15万像素的轨道。
- 主题轨道的`requestAnimationFrame`永久运行，每帧读取布局；页面不可见或模块离开视口后仍不停止。

## 方案

### 固定卡槽双轨

- 每行保留12个逻辑卡槽，并复制一次形成无缝轨道。
- 两行合计48张影像卡，而不是720张。
- 完整影像目录以紧凑JSON数据存在页面中，不提前创建图片和卡片节点。
- 每次轨道完成一轮后切换到下一批12条，两个副本同时更新。
- 初始批次随机选择，避免每次打开都从同一作品开始。
- 两行继续向相反方向运动，手机、平板和桌面保持同一种视觉语言。

### 轨道生命周期

- 影像区进入视口后才启动轨道。
- 影像区离开视口、页面进入后台或浏览器开启“减少动态效果”时暂停。
- 精确指针设备悬停轨道时暂停，触屏设备不依赖hover。
- 手机和平板不再使用`animation: none`和手动超长滚动轨道。

### 交互收敛

- 影像详情的双击和键盘事件改为轨道级事件委托。
- 卡片倾斜只在`(hover: hover) and (pointer: fine)`设备启用。
- 动态换批后的卡片不注册独立监听器。
- 近期在看、近期看完、雷达图等少量独立控件保持原交互。

### 图片约束

- 轨道图片添加固定`width`、`height`，减少布局抖动。
- 使用`loading="lazy"`、`decoding="async"`和`fetchpriority="low"`。
- 当前阶段继续使用原图片地址，不引入图片处理依赖。
- WebP/AVIF缩略图及上传时自动生成缩略图作为后续独立阶段。

### RAF治理

- 主题轨道只在模块可见且页面位于前台时请求下一帧。
- 使用`ResizeObserver`缓存轨道缩放比例，动画帧内不再读取布局。
- 离开视口后取消RAF，重新进入时恢复。
- 有限时长的数字计数动画维持现状。

## 性能预算

在390×844和768×1024视口下：

- `.qzq-watch-card`不超过48个。
- 首页图片节点不超过70个。
- 首页DOM元素不超过1800个。
- 每条影像轨道宽度不超过16000px。
- 影像区离开视口或页面隐藏时，轨道动画状态为`paused`。
- 页面级`scrollWidth`等于`clientWidth`。

## 文件边界

- `src/pages/index.astro`：从现有完整行数据生成初始随机批次，渲染固定卡槽、嵌入紧凑目录，并实现换批、事件委托和动画生命周期。
- `src/styles/global.css`：恢复移动端轻量双轨，统一暂停状态，移除移动端超长手动轨道规则和不必要的长期合成提示。
- `tests/homePerformanceSource.test.mjs`：锁定卡槽数量、事件委托、可见性暂停和图片属性。

## 验证

- 新测试先在旧实现上失败，再实现修复。
- 运行`node --test`、`npm run build`和`git diff --check`。
- 使用真实浏览器在390×844、768×1024和1440×1000测量DOM、图片、卡片、轨道宽度和页面溢出。
- 截图检查两排反向轨道、卡片比例、统计区衔接和触屏布局。
- 不在本阶段修改Nginx；服务器部署后单独开启gzip或Brotli。
