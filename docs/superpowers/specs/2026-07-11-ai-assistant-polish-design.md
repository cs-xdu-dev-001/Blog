# AI助手体验收尾设计

## 目标

在现有可拖动、可缩放的AI助手窗口上补齐连续对话能力，让它具备完整但克制的聊天体验：理解最近对话、可清空、可取消，并安全显示Markdown回答。

## 范围

- 客户端仅在当前页面内保存最近6轮对话，共最多12条历史消息。
- 关闭再打开窗口时保留对话，刷新页面后清空。
- 聊天记录不写入localStorage、sessionStorage、SQLite或服务端会话。
- 不增加流式输出、跨设备同步、会话列表和历史检索。

## 交互设计

### 连续对话

每次发送时，客户端把已完成的最近6轮历史和当前问题一起交给API。当前问题仍单独用于检索本站内容，历史只帮助模型理解指代和语境。

生成失败、限流或主动取消产生的提示不进入后续模型上下文。只保留真实用户消息与成功生成的AI回答。

### 清空对话

标题栏增加“清空”命令。没有消息时按钮禁用；有消息时点击立即清空并恢复欢迎语。若正在生成，先取消当前请求再清空，不弹二次确认。

### 取消生成

正常状态下表单按钮为“发送”；生成时同一位置变为“停止”，避免增加新的常驻控件。点击后通过`AbortController`取消请求，未完成的AI消息显示“已停止生成”，随后恢复输入能力。

### Markdown回答

AI消息支持段落、标题、无序和有序列表、引用、粗体、斜体、链接、行内代码和围栏代码块。原始HTML一律转义，链接仅允许安全协议，外部链接补充`target="_blank"`和`rel="noopener noreferrer"`。

用户消息仍按纯文本显示，避免把输入误渲染成界面结构。

## 数据协议

请求体调整为：

```json
{
  "question": "当前问题",
  "messages": [
    { "role": "user", "content": "上一轮问题" },
    { "role": "assistant", "content": "上一轮回答" }
  ]
}
```

服务端不信任客户端历史：只接受`user`和`assistant`角色，清理空白，限制单条长度、总条数和总字符数，并取最后12条。`system`、`tool`及未知角色全部丢弃。

Chat Completions模式把规范化历史插入system消息和当前问题之间；Responses模式转换为对应的`input_text`结构。本站检索上下文只附加到当前问题，避免在每轮历史中重复膨胀。

## 代码边界

- `public/assistant.js`：窗口交互、消息状态、清空、取消和DOM更新。
- 可测试的纯函数模块：历史裁剪、安全Markdown渲染和URL校验。
- `src/pages/api/assistant.ts`：读取并转交`messages`。
- `src/lib/server/assistantService.mjs`：服务端历史规范化及模型请求组装。
- `src/styles/global.css`：标题栏命令、停止状态和Markdown排版。

不改数据库结构，不改管理端配置，不重构首页其他模块。

## 错误与状态

- 主动取消：中性提示“已停止生成”，不视为错误。
- 网络失败：当前AI消息显示连接失败，保留用户问题。
- 接口错误或限流：显示后端返回信息并使用错误样式。
- 清空：终止当前请求、清除内存历史和消息DOM、恢复欢迎语。

## 测试与验收

- 服务端历史最多保留最近12条，并过滤非法角色、空内容和超长内容。
- Chat Completions与Responses两种请求均按正确顺序携带历史。
- Markdown渲染覆盖标题、列表、引用、代码、链接，并验证脚本和危险URL不能执行。
- 前端状态验证清空后历史归零，取消后不把未完成回答写入上下文。
- 运行现有AI助手测试、JavaScript语法检查和Astro生产构建。
- 浏览器检查发送、停止、清空、连续追问、代码块和窗口缩放下的排版。
