前端页面最容易出现一种错觉：代码看起来合理，浏览器里却不对。尤其是博客、作品集和仪表盘这类重排版页面，问题经常不是功能错误，而是间距、断行、对齐和层级不稳。

截图验证的价值，就是把“我觉得应该没问题”变成可观察的证据。

## 代码不能替代视觉检查

CSS 是约束系统，不是静态图片。标题长度、中文断行、视口宽度、字体渲染、滚动条和浏览器默认样式都会影响最终结果。

只看组件代码，很容易忽略这些问题：移动端出现横向滚动，按钮文字挤在一起，标题换行后节奏断掉，列表列宽在真实内容下失衡。

## 至少检查两个视口

一个页面最少应该看桌面和手机两个视口。桌面可以用 1440px 宽度，手机可以用 375px 宽度。前者看整体版式和留白，后者看信息是否还能读。

检查点很明确：

- `scrollWidth` 是否等于 `innerWidth`。
- 首屏是否能说明页面是什么。
- 长标题和中文段落是否自然换行。
- 导航和按钮是否有足够点击面积。
- 内容区是否过宽或过窄。

这些检查不复杂，但能拦住很多“看起来像错版”的问题。

## 截图要配合真实内容

空数据和短标题会掩盖布局问题。博客尤其需要用真实标题、真实摘要和真实正文测试。

如果所有文章标题都只有四个字，列表当然整齐；真正上线后出现十几个字的中文标题、英文术语和日期信息，网格才会暴露问题。所以视觉验证应该尽早使用接近真实的内容。

## 不要过度依赖动画

动效可以帮助层级，但也会遮住问题。页面刚加载时如果标题还在淡入，截图可能看起来发灰，用户也可能误以为内容没加载完。

对阅读型页面来说，最稳定的交互通常很克制：链接 hover 改变透明度，列表边框变深，焦点状态清晰可见。动画应该服务阅读，而不是抢走注意力。

## 把验证放进工作流

一个实用流程是：

1. 改完页面后运行构建。
2. 用浏览器打开本地地址。
3. 截桌面和手机两张图。
4. 检查横向溢出和首屏信息。
5. 根据截图做小步调整。

这套流程不会让设计自动变好，但能避免很多明显错误。UI 不是写完 CSS 就结束，必须回到真实渲染结果里校准。
